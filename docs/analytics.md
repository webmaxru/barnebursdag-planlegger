# Analytics, privacy & the user-engagement dashboard

Kakeklar uses **Azure Application Insights** to track key engagement events — configured in a
**cookieless** way so that **no cookie-consent banner is legally required**.

## Why there is no cookie banner

The EU ePrivacy Directive and Norway's **ekomloven §3-15** (in force 1 Jan 2025) require prior
consent only when a site **stores or reads information on the user's device** (cookies, localStorage)
for non‑essential purposes. Datatilsynet's guidance treats analytics that set **no cookies and no
persistent identifiers**, processing only aggregated/anonymous data, as **not** requiring a consent
banner (the same basis on which tools like Plausible operate).

Kakeklar's analytics is configured to meet that bar, so **adding a banner is not required** — which
is exactly what the brief asked for ("only if required by EU law"). Concretely:

| Setting | Effect |
|--------|--------|
| `disableCookiesUsage: true` | Application Insights sets **no cookies** (`ai_user`, `ai_session`). |
| `enableSessionStorageBuffer: false` | No sessionStorage buffer → **zero device storage**. |
| No persistent user id | User/session ids are in‑memory only, regenerated each load (no cross‑visit tracking). |
| Default IP masking | App Insights derives coarse geo, then stores `client_IP = 0.0.0.0`. |
| `disableAjaxTracking` / `disableFetchTracking` | We don't auto‑capture URLs of XHR/fetch calls. |

Net result: anonymous, aggregate usage analytics with nothing stored on the device → no banner.
See [`src/lib/analytics.ts`](../src/lib/analytics.ts).

## How it's wired

```mermaid
graph LR
  B["Browser"] -->|GET /api/config| S["Express server"]
  S -->|connectionString (from ACA secret)| B
  B -->|lazy import SDK| C["@microsoft/applicationinsights-web (separate chunk)"]
  C -->|cookieless events| AI["Application Insights (kakeklar-insights)"]
  AI --> WB["Workbook: Kakeklar – User Engagement"]
```

- The connection string is **never bundled**. The server exposes it at runtime via `GET /api/config`
  (sourced from the ACA secret `appinsights-connection-string` / env
  `APPLICATIONINSIGHTS_CONNECTION_STRING`). Locally, with no key, `/api/config` returns
  `{ appInsights: null }` and analytics is **disabled gracefully**.
- The SDK is **code-split** (lazy `import()`), so it never blocks first paint — the main bundle stays
  ~55 kB gzip; the ~77 kB gzip SDK chunk loads after render.

## Event catalog

`track(name, properties)` in [`src/lib/analytics.ts`](../src/lib/analytics.ts) sends these custom events:

| Event | Fires when | Properties |
|-------|-----------|------------|
| *(pageView)* | App initialises | name `Kakeklar` |
| `wizard_started` | Wizard starts | – |
| `wizard_step` | Wizard step changes | `step` |
| `wizard_completed` | Wizard completes | – |
| `wizard_skipped` | Wizard is skipped | – |
| `party_configured` | 1.2 s after guests/age/type/duration change (debounced) | `guests`, `age`, `partyType`, `durationHours` |
| `party_type_changed` | Home ↔ kindergarten toggle | `type` |
| `allergy_changed` | An allergy/restriction "antall personer" slider changes | `allergy`, `count` |
| `plan_shared` | Del / share | `method` (`webshare`/`clipboard`), `guests`, `age` |
| `plan_printed` | Skriv ut | `guests`, `age` |
| `config_opened` | Tilpass opened | – |
| `config_changed` | Catalog edited | `action` (`add`/`remove`/`reset`/`import`/`export`) |
| `price_lookup` | "Sjekk pris" (Kassal) | `item` |
| `pwa_installed` | App installed to home screen | – |

Numeric properties are also sent as App Insights *measurements* for aggregation.

## The dashboard (Azure Monitor Workbook)

A workbook **"Kakeklar – User Engagement"** is deployed to `rg-barnebursdag` and linked to the
`kakeklar-insights` component. Open it in the portal: **Application Insights → kakeklar-insights →
Workbooks → Kakeklar – User Engagement** (or Azure Monitor → Workbooks).

It shows, with a time-range pill:
- **Engagement over time** — daily active sessions, events per day, page views.
- **What users do** — top events, key actions (share/print/config/price lookup), share method.
- **Which parties get planned** — distribution of child age, guest count, and home-vs-kindergarten split.

The workbook is defined as code in [`infra/engagement-workbook.json`](../infra/engagement-workbook.json)
(generated by `infra/gen-workbook.mjs`). Redeploy with:

```bash
node infra/gen-workbook.mjs
az deployment group create -g rg-barnebursdag \
  --template-file infra/engagement-workbook.json \
  --parameters appInsightsResourceId="$(az monitor app-insights component show --app kakeklar-insights -g rg-barnebursdag --query id -o tsv)"
```

### The portal dashboard (Microsoft.Portal/dashboards)

A pinnable Azure **dashboard** "Kakeklar – Key Metrics" (`kakeklar-key-metrics`) is deployed to
`rg-barnebursdag`. Open it directly:
**[portal.azure.com → Kakeklar – Key Metrics](https://portal.azure.com/#@/dashboard/arm/subscriptions/d0b7d6ee-17bf-4c4f-b79d-4f6c2cb583fd/resourceGroups/rg-barnebursdag/providers/Microsoft.Portal/dashboards/kakeklar-key-metrics)**
(or Azure portal → **Dashboard** → Browse → *Kakeklar – Key Metrics*).

Nine tiles: visitors/traffic over time, engagement events per day, top events, key actions, the
wizard funnel, child-age distribution, guest-count distribution, and the home-vs-barnehage split.
It is the same data as the workbook, in an at-a-glance pinned layout.

Defined as code in [`infra/dashboard.json`](../infra/dashboard.json). Redeploy with:

```bash
az deployment group create -g rg-barnebursdag --template-file infra/dashboard.json
```

### Handy KQL (Logs blade)

```kusto
// Engagement: events per day
customEvents | summarize count() by bin(timestamp, 1d) | render columnchart

// Age distribution of planned parties
customEvents
| where name == 'party_configured'
| extend age = toint(customDimensions.age)
| summarize parties = count() by age | sort by age asc

// Conversion-style actions
customEvents
| where name in ('plan_shared','plan_printed','config_opened','price_lookup')
| summarize count() by name
```

## Pull metrics from the terminal

Don't want to open the portal? Run the bundled script to print the key metrics straight in your shell:

```powershell
npm run insights                              # last 30 days (default)
pwsh -NoProfile -File ./scripts/insights.ps1 -Days 7
```

It prints **visitors & traffic, engagement events/day, top events, key actions, the wizard funnel,**
and **child-age / guest-count / home-vs-barnehage distributions** — sourced from Application Insights
via `az monitor app-insights query`. The script auto-installs the `application-insights` CLI
extension, resolves the app id from `kakeklar-insights` (override with `-AppId`/`-Component`/
`-ResourceGroup`), and is empty-safe (shows `(no data yet)` where there's nothing). Requires
`az login`. Source: [`scripts/insights.ps1`](../scripts/insights.ps1).

## Turning analytics off

Remove/empty the `APPLICATIONINSIGHTS_CONNECTION_STRING` env var (or the ACA secret). The client then
receives `null` from `/api/config` and never loads the SDK — the app works identically, just without
telemetry. No code change required.
